# ADR: `a2a-team` — Multi-Agent Team Coordination Layer | (**status:** `superseded`)

**SUPERSEDED** — This ADR is fully superseded by
`[[2026-02-24-subagent-protocol-adr]]` (Unified A2A Protocol Stack — Full
Rewrite). The assumption that ACP bridges remain alongside A2A team layer is
eliminated. Teams now use the same unified A2A stack as subagents. Do not
implement any decisions from this document.

## Status

**Approved** — All 6 architectural decisions confirmed interactively.

## Context

The existing A2A implementation (ADR `2026-02-15`, Phases 1–5 complete) provides
bilateral agent communication: one coordinator dispatches to one agent server at a
time. The `a2a-team` feature extends this to N-agent team coordination — named teams
with multiple members, parallel task dispatch, two-way communication, coordinated
lifecycle, and feature parity between Claude and Gemini participants.

The Claude Code runtime already provides native team tooling (TeamCreate, SendMessage,
TaskCreate). The A2A protocol is the common ground that achieves feature parity: Claude
accesses it via its native team tools; Gemini reaches it via the experimental `@a2a`
client tool. A2A is the equalizer.

______________________________________________________________________

## Decision 1 — TeamCoordinator lives in `orchestration/team.py`

**Decision:** Create `orchestration/team.py` as a new module alongside
`task_engine.py` and `subagent.py`. The `TeamCoordinator` and `TeamSession` are
first-class siblings of `TaskEngine` — same import patterns, same testing conventions.

**Rationale:** The team orchestration concern (multi-agent lifecycle, parallel
dispatch, result aggregation) is distinct from single-task tracking (TaskEngine) and
single-agent dispatch (subagent.py). Co-location in `orchestration/` groups all
agent-related coordination. Placing it inside `protocol/a2a/` would couple team logic
to the A2A transport layer, breaking the layered architecture.

**Core types:**

```python

# orchestration/team.py

@dataclass
class TeamMember:
    name: str
    url: str
    card: AgentCard
    status: MemberStatus   # spawning|idle|working|shutdown_requested|terminated

class TeamStatus(StrEnum):
    FORMING    = "forming"
    ACTIVE     = "active"
    DISSOLVING = "dissolving"
    DISSOLVED  = "dissolved"

@dataclass
class TeamSession:
    team_id: str      # = context_id (see Decision 2)
    name: str         # human-readable, forwarded in metadata["team_name"]
    context_id: str   # A2A session token (= team_id)
    members: dict[str, TeamMember]
    status: TeamStatus
    created_at: float

class TeamCoordinator:
    async def form_team(name, agent_urls, api_key=None) -> TeamSession
    async def dispatch_parallel(assignments: dict[str, str]) -> dict[str, Task]
    async def relay_output(src_task: Task, dst_agent: str, instructions: str) -> Task
    async def collect_results() -> dict[str, str]
    async def ping_agents() -> dict[str, bool]
    async def dissolve_team() -> None
```

## Decision 2 — `contextId` IS the `team_id`; name forwarded in metadata

**Decision:** The coordinator generates a single UUID at team formation. This UUID
serves as both the vaultspec `team_id` and the A2A `contextId` sent to every agent
server. The human-readable team name is stored on `TeamSession` and forwarded in
`metadata["team_name"]` on every `SendMessageRequest`.

**Rationale:** A2A's `contextId` is designed to group related tasks into a shared
session — exactly the semantics of a team session. Using the same value for both
eliminates a mapping table and makes `ListTasks(context_id=team_id)` a natural way to
enumerate all tasks in a team. A2A has no native name field; `metadata` is the spec's
intended extension point for such annotations.

```python
team_id = str(uuid4())   # "a3f9-..."
context_id = team_id     # identical

# Every outbound message:

Message(
    context_id=team_id,
    metadata={"team_name": "my-research-team"},
    reference_task_ids=[...],
    parts=[TextPart(text=task_text)],
)
```

## Decision 3 — Gemini: A2A server wrapper + experimental `@a2a` client

**Decision:** Gemini participates in teams via two mechanisms:

1. **As an A2A server target**: The existing `GeminiA2AExecutor` wraps Gemini CLI into
   an A2A-compliant HTTP endpoint. The HTTP server process persists; each inbound task
   spawns a fresh Gemini CLI subprocess via ACP. The coordinator connects as `A2AClient`.

1. **As an experimental A2A client**: With `experimental.enableAgents: true` in
   `~/.gemini/settings.json`, Gemini can sub-delegate tasks to other A2A agents during
   its own execution via the `@a2a` tool. This is enabled by the existing `discovery.py`
   module which generates `.gemini/agents/<name>.md` config files pointing to vaultspec
   A2A servers.

**Rationale:** Feature parity is the goal. Claude participates in teams via its native
runtime tools (TeamCreate/SendMessage). Gemini's equivalent is A2A: it acts as a server
target (receiving tasks from the coordinator) and optionally as an A2A client
(sub-delegating during execution). A2A is the protocol equalizer. The
`experimental.enableAgents` path is explicitly supported and leveraged, not avoided.

```

# Local dev team:

Claude Code (coordinator)
  |-- A2AClient --> ClaudeA2AExecutor  :10010  (Claude agent server)
  |-- A2AClient --> GeminiA2AExecutor  :10011  (Gemini agent server)

# Gemini sub-delegates during execution (optional):

GeminiA2AExecutor :10011
  |-- @a2a tool --> ClaudeA2AExecutor :10010
```

**Constraint:** Gemini cannot be a natively persistent lingering team member — each
task creates a new subprocess. The A2A server wrapper is the persistence layer.

## Decision 4 — CLI: new `lib/scripts/team.py` script

**Decision:** Expose team lifecycle as a standalone `team.py` CLI script, following the
same conventions as `subagent.py`, `vault.py`, and `cli.py` exactly:

- `--root` flag (`type=Path`, `default=None`, `ROOT_DIR` env fallback)
- `get_config()` from `core.config` for framework path resolution
- `argparse` subcommands (not typer/click)
- Quiet by default; `-v` / `--verbose` for detail

**Commands:**

```
python team.py create  --name NAME --agents agent1:port1[,agent2:port2,...] [--api-key KEY]
python team.py status  --name NAME
python team.py assign  --name NAME --agent AGENT --task "description"
python team.py broadcast --name NAME --message "text"
python team.py message --name NAME --to AGENT --content "text"
python team.py dissolve --name NAME [--force]
python team.py list
```

**Rationale:** Team lifecycle is a distinct concern from single-agent dispatch
(subagent.py). A separate script keeps each entry point focused, avoids bloating
subagent.py, and maintains the project convention of one script per domain. CLI
uniformity is explicitly required.

## Decision 5 — Cross-agent result relay: coordinator-mediated injection

**Decision:** When Agent B's task depends on Agent A's output, the coordinator mediates
the relay: it fetches A's completed task artifacts via `A2AClient.get_task()`, extracts
the text content, and injects it into B's incoming message body alongside
`reference_task_ids: [A's task ID]`.

**Rationale:** A2A agent servers are opaque black boxes — B's server cannot query A's
task store directly. The coordinator is the only entity with `A2AClient` connections
to all servers. This pattern requires no shared infrastructure, works with the default
`InMemoryTaskStore`, and preserves the A2A protocol's design intent. It is already
demonstrated by the French novel relay test (`test_french_novel_relay.py`).

```python
async def relay_output(
    self, src_task: Task, dst_agent: str, instructions: str
) -> Task:
    text = extract_artifact_text(src_task)
    return await self._dispatch_one(
        dst_agent,
        SendMessageRequest(params=MessageSendParams(
            message=Message(
                context_id=self.session.team_id,
                metadata={"team_name": self.session.name},
                reference_task_ids=[src_task.id],
                parts=[
                    TextPart(text=text),          # A's output
                    TextPart(text=instructions),  # B's task
                ],
            )
        )),
    )
```

## Decision 6 — Security: unauthenticated localhost; API key opt-in

**Decision:** Local team sessions run without authentication (the A2A spec permits
unauthenticated connections). Production deployments opt in via `--api-key` on
`team.py create`; the coordinator passes the key as `X-API-Key` on all outbound
requests and agent servers declare the scheme in their Agent Cards.

Model-level authentication (Gemini OAuth/API key, Claude ANTHROPIC_API_KEY) is handled
separately by the existing provider infrastructure and is not the concern of
team-level transport security.

```python

# Local: no auth

AgentCard(security_schemes={}, security=[])

# Production: opt-in API key

AgentCard(
    security_schemes={
        "teamApiKey": APIKeySecurityScheme(
            name="X-API-Key", location="header"
        )
    },
    security=[{"teamApiKey": []}],
)
```

**Rationale:** Both Claude and Gemini handle model authentication internally (Gemini via
Google OAuth/GEMINI_API_KEY; Claude via ANTHROPIC_API_KEY). Team-level transport auth
is a separate concern — an optional envelope over already-authenticated agent processes.
Zero-friction local dev is a hard requirement; the opt-in model satisfies both.
