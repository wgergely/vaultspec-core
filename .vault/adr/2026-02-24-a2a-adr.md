> This ADR is **binding**. It supersedes all previous protocol-layer ADRs
> listed below. Any future work on the subagent or team protocol layer MUST
> conform to the decisions in this document. Deviations require a new ADR
> that explicitly supersedes this one.

Every superseded ADR assumed ACP bridges and A2A servers coexist as parallel
protocol stacks. This ADR eliminates that duality.

## Problem Statement

The protocol layer maintains **two parallel stacks** for the same logical
operation (spawn agent → send prompt → get result):

- `acp/` — ACP bridges (`claude_bridge.py`, `gemini_bridge.py`,
  `client.py`, `types.py`) for subagents via stdio.
- `a2a/` — executors, server, and agent cards for team coordination via HTTP.

Both need session management, cancellation, retry, and streaming. Every fix
is implemented twice, tested twice, and maintained twice. 10 prior ADRs
attempted incremental fixes without solving the root cause: two stacks doing
the same thing.

## Considerations

- **ACP semantic mismatch** — ACP is designed for editor↔agent integration
  (sessions, modes, terminals). Using it for orchestrator↔subagent comms
  forces 15-method `Agent` entries where only 3 carry real logic.
- **A2A semantic fit** — A2A's task model (submit → working → completed)
  maps directly to vaultspec's interaction pattern.
- **Salvage vs rewrite** — ACP bridge code is structurally coupled to the
  wrong protocol. A 15-method class cannot be "refactored" into a 2-method
  class. `gemini_bridge.py` already had 4 CRITICAL issues and was scheduled
  for full rewrite.
- **Claude SDK** — `ClaudeA2AExecutor` already uses `claude-agent-sdk`
  directly without ACP. No rewiring needed (see
  `[[2026-02-24-cli-protocols-research]]`).
- **Gemini CLI** — no native A2A server mode. The executor will continue to
  wrap `gemini --experimental-acp` internally while exposing A2A externally
  (see `[[2026-02-24-cli-protocols-research]]`).

## Constraints

### Protocol role assignment (final)

| Protocol | Role | Library | Status |
| -------- | ---- | ------- | ------ |
| **A2A** | Agent↔agent (subagents AND teams) | `a2a-sdk` | **sole protocol** |
| **MCP** | Agent↔tool (vault search, team ops) | `mcp` | unchanged |
| **ACP** | Editor↔agent (Zed integration only) | `acp` | editor-only |

| Gemini ACP bridge | `protocol/acp/gemini_bridge.py` | ~383 |
| ACP client | `protocol/acp/client.py` | ~500 |
| ACP types | `protocol/acp/types.py` | ~100 |
| ACP `__init__` | `protocol/acp/__init__.py` | ~20 |

> [!WARNING]
> The `acp/` directory under `protocol/` is deleted entirely. The
> `agent-client-protocol` pip package remains installed for Zed editor
> integration only. NO new code in `src/vaultspec/protocol/` may import
> from `acp`.

### What is preserved from the existing A2A layer

- `executors/claude_executor.py` — functional, needs hardening
- `executors/gemini_executor.py` — functional, needs hardening
- `server.py` — A2A Starlette application, usable as-is
- `agent_card.py` — agent card generation, usable as-is
- `discovery.py` — agent discovery, usable as-is

## Implementation

A2A becomes the **only** agent communication protocol. Both subagent
spawning and team coordination use the same A2A-over-localhost-HTTP stack.

- **Subagent spawning** uses `AgentExecutor` → `A2AStarletteApplication` →
  `A2AClient`, same as team coordination.
- **One set of hardening** (retry, cancel, streaming, session resume)
  implemented once in shared base classes.
- **One test infrastructure** — `respx` HTTP mocking, A2A TCK conformance.

## Rationale

1. **~3500 LOC net reduction** — ~3800 deleted, ~300 added.
2. **One protocol stack** — every hardening fix is implemented, tested,
   and maintained once.
3. **A2A Inspector** — web-based JSON-RPC debug console replaces opaque
   stdio pipes.
4. **`respx` testing** — HTTP-layer mocking replaces fragile subprocess
   mocking.
5. **Standards alignment** — A2A is a Linux Foundation standard (AAIF)
   with active development. ACP for subagents was a misuse of an
   editor-oriented protocol.

## Consequences

- **Positive:** Single code path, inspector-debuggable, simplified
  testing, team–subagent parity.
- **Negative:** ACP bridge investment (~3800 LOC) is discarded.
  `ServerProcessManager` is new critical code that must be thoroughly
  tested. Orphan risk requires parent-PID watchdog (mitigated in
  `[[2026-02-24-subagent-protocol-server-manager-adr]]`).
